# AGENTS.md
## 1. 目的
本文件用于约束 Codex 在本项目中的编码行为。
最高优先级要求：
- 代码按职责分模块；
- 模块之间保持低耦合；
- 新功能便于扩展和替换；
- 关键代码使用简洁、准确的中文注释；
- 每个新建 Python 文件顶部必须写中文职责说明；
- 修改范围尽可能小；
- 不得为了未来不确定需求过度设计。
本文件不固定最终目录结构，项目结构应随真实需求逐步演进。

## 2. 开始任务前
Codex 开始编码前必须：
1. 阅读本文件；
2. 检查 `git status`；
3. 了解当前目录结构；
4. 阅读相关文件、调用方、依赖和测试；
5. 明确本次修改涉及的模块边界；
6. 选择改动最小、风险最低的方案。
不得：
- 覆盖用户未提交的修改；
- 修改与任务无关的代码；
- 在不了解现有实现时直接重写；
- 擅自执行 `git push`；
- 擅自修改生产配置或真实凭证；
- 声称未实际运行的检查已经通过。

## 3. 模块划分
每个模块、文件和类都应有清晰且相对单一的职责。
一个文件应围绕一个明确主题组织代码，例如模型调用、工具注册、工具执行、记忆管理、状态管理、配置管理、数据模型和外部平台适配。
不得把配置读取、模型调用、数据库访问、业务逻辑和接口处理全部堆在同一个文件或类中。
是否拆分模块应根据职责和变化原因判断，不能只根据文件行数判断。
当一个文件同时存在多个独立职责、难以测试或经常因不同原因修改时，应进行拆分。
拆分后必须同步更新导入、调用方、测试和相关文档。

## 4. 单一职责
一个类不应同时负责多个无关领域。
一个函数应尽量只完成一个明确任务。
函数过长时，应按照业务步骤提取小函数，并使用能够表达业务含义的名称。
优先使用 `build_context()`、`validate_tool_call()`、`execute_tool()`、`save_memory()` 等明确命名。
避免使用 `process1()`、`handle_data()`、`do_work()` 等含义模糊的名称。

## 5. 解耦原则
高层业务逻辑不得直接依赖具体第三方实现。
Agent 核心逻辑不应写死依赖某个模型供应商、数据库或外部平台。
依赖应优先通过构造参数注入。
具体实现的创建和组装应集中在程序入口或依赖装配模块。
业务类只使用依赖，不负责决定依赖的具体实现。
模块之间只能通过公开接口和明确的数据模型交互。
禁止跨模块直接访问私有属性、内部缓存、私有数据库连接和第三方 SDK 内部对象。
必须避免循环依赖。出现循环依赖时，应重新检查模块职责和依赖方向。

## 6. 外部实现隔离
模型 API、数据库、文件系统和其他第三方平台必须与核心业务逻辑隔离。
第三方 SDK 的专有类型不得扩散到核心业务模块。
应在模块边界处，把第三方数据转换为项目自己的数据模型。
新增外部实现时，应优先增加新的适配器或实现类，而不是修改核心业务逻辑。

## 7. 接口与扩展
存在多个实现、测试需要 Mock、底层实现可能替换或需要明确调用契约时，可以定义抽象基类或 `Protocol`。
公开接口应使用明确的业务类型，避免长期传递结构不固定的 `dict`。
新增模型供应商、记忆存储、工具类型或平台适配时，应优先通过新增实现完成扩展。
避免在核心逻辑中不断增加 `if/elif`。
当类型持续增加时，可使用注册表、策略对象、映射表或独立处理器。
如果当前逻辑很简单，可以保留简单实现，不得为了设计模式而设计。

## 8. 项目结构演进
只创建当前任务真正需要的文件和模块。
不得提前创建：
- 大量空目录；
- 只有 `pass` 的空类；
- 没有调用方的接口；
- 只做简单转发的多层包装；
- 为未来不确定需求准备的复杂架构。
真正的可扩展性来自职责清晰、依赖方向正确、接口稳定、数据模型明确和测试完善。

## 9. 数据模型
模块之间传递复杂数据时，应使用明确的数据模型。
可根据实际情况使用 `dataclass`、Pydantic、`Enum`、`TypedDict` 或 `Protocol`。
字段和变量名称必须表达真实业务含义。
涉及单位时，应在名称中体现，例如 `timeout_s`、`duration_ms`、`file_size_bytes`、`timestamp_utc`。
避免使用 `data`、`temp`、`obj`、`value` 等含义不清的名称。

## 10. 中文注释
Codex 编写的注释和文档字符串必须使用中文。
注释应简洁，重点说明：
- 文件、类或函数的职责；
- 关键业务规则；
- 模块边界；
- 为什么采用当前实现；
- 容易误解的逻辑；
- 第三方 SDK 的特殊限制；
- 异步、并发和资源释放要求。
不要机械翻译每一行代码。
简单赋值、循环和直观条件判断不需要添加噪音注释。

## 11. 文件顶部职责说明
每个新建 Python 文件都必须在顶部使用中文模块文档字符串说明文件职责。
推荐格式：
```python
"""
本文件负责 Agent 工具的注册与查找，并向模型提供工具定义。
本文件不负责执行工具，也不直接调用第三方平台。
"""
```
职责较复杂时，可以补充主要功能和职责边界，但不必写成长篇说明。
修改已有 Python 文件时，如果缺少顶部职责说明，应在不影响现有结构的前提下补充。
文件职责发生变化时，必须同步更新顶部说明。
禁止使用 `"""这是一个工具文件。"""` 等无意义说明。
JSON、自动生成文件和第三方代码不强制添加该说明。

## 12. 类、函数和行内注释
公共类、公共函数、抽象接口和核心业务函数应添加简洁的中文文档字符串。
文档字符串根据需要说明职责、参数、返回值、异常和调用限制。
行内注释应解释“为什么”，而不是重复代码做了什么。
禁止使用“特殊处理”“临时处理”等含糊注释，必须说明具体原因。
代码修改后，应同步检查旧注释是否仍然准确。

## 13. 异常、测试和修改范围
不得使用裸 `except`，不得静默吞掉异常。
只有真正能够处理异常的层才应捕获异常。
包装异常时，应保留原始异常上下文。
核心业务逻辑、数据转换、状态变化和外部适配应尽量能够独立测试。
单元测试不得依赖真实生产服务，应使用 Mock、Fake 或测试适配器隔离外部依赖。
修改代码时必须：
- 保持范围尽可能小；
- 不修改无关文件；
- 不随意改变公共接口；
- 不进行无关的全项目格式化；
- 公共接口变化时检查所有调用方；
- 不删除失败测试来让测试通过。

## 14. 完成任务后的自检
完成编码后，Codex 必须检查：
1. 模块职责是否清晰；
2. 是否存在不必要的强耦合；
3. 外部实现是否与核心逻辑隔离；
4. 是否存在循环依赖；
5. 是否存在无意义抽象或空架构；
6. 新增 Python 文件是否有中文职责说明；
7. 关键逻辑是否有必要的中文注释；
8. 注释是否与当前代码一致；
9. 是否修改了无关文件；
10. 是否补充并实际运行了相关测试。
最终回复必须使用中文，说明修改内容、模块划分与解耦方式、后续扩展点、实际运行的检查以及未验证内容或风险。
